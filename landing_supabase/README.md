# راهنمای استقرار و معماری ماژول مارکتینگ و اتریبیوشن هکسر (Hexer Marketing Hub)

این دایرکتوری شامل ساختار، کدهای پایگاه‌داده و شیوه‌نامه راه‌اندازی ماژول تحلیلی و سیستم ردیابی ترافیک بیرونی پلتفرم هکسر است.

---

## ۱. نمای کلی سیستم و معماری داده
داده‌های پنل بر مبنای الگوی **Multi-Database Federated Architecture** طراحی شده است تا ترافیک سنگین کلاینت‌ها و رویدادهای مارکتینگ (Landing Page Views/Clicks) تاثیری بر پایگاه داده عملیاتی کلاینت اصلی ایجاد نکند:

1. **پروژه Supabase تحلیلی (Remote Analytics DB):**
   - میزبان داده‌های خام کمپین‌ها و کلیک‌ها ترافیکی با هدف حفظ کارایی و عدم انباشت حجم گاوصندوق ابری.
   - دسترسی نوشتن عمومی (`anon` INSERT) صرفاً روی جدول `events`.

2. **پروژه Supabase اصلی (Main Production DB):**
   - دارای پل بریج FDW اختصاصی جهت وارد کردن نماهای تحلیلی ریموت به محدوده لوکال.
   - دارای ۶ نمای ماتریالیزه شده عمیق جهت گزارش‌گیری با سرعت و کارایی بالا (تثبیت‌شده در قالب Materialized Views).
   - رفرش هوشمند زمینی بک‌گراند توسط `pg_cron` مستقل هر ۳۰ دقیقه یکبار.

---

## ۲. ترتیب اجرای اسکریپت‌های پایگاه داده (زنجیره مهاجرت)

برای استقرار موفقیت‌آمیز فاز مارکتینگ، اسکریپت‌ها را دقیقاً به ترتیب زیر بر روی پایگاه‌داده‌های مربوطه اجرا نمایید:

### گام اول: آماده‌سازی دیتابیس تحلیلی (ریموت)
فایل زیر را روی **پروژه سوپابیس تحلیلی جدید** اجرا کنید:
1. `landing_supabase/sql/01_analytics_schema.sql`
   - ایجاد پایه‌های اسکیما (`events` و `campaigns`).
   - ساخت نقش دسترسی سفارشیِ بی‌آزار `mkt_bridge_user` با دسترسی‌های غیر سوپریوزر و تعریف RLS سفارشی.

### گام دوم: ثبت گاوصندوق امنیتی در دیتابیس اصلی (Vault)
ابتدا متغیرهای اتصال دیتابیس تحلیلی را بصورت رکورد در گاوصندوق امنیتی سوپابیس اصلی وارد نمایید تا پل ارتباطی FDW بوجود آید:
```sql
-- مقداردهی به متغیرهای ورود در دیتابیس اصلی جهت اتصال FDW
SELECT vault.create_secret('YOUR_REMOTE_ANALYTICS_HOST', 'analytics_db_host');
SELECT vault.create_secret('5432', 'analytics_db_port');
SELECT vault.create_secret('postgres', 'analytics_db_name');
SELECT vault.create_secret('mkt_bridge_user', 'analytics_db_user');
SELECT vault.create_secret('YOUR_REMOTE_ANALYTICS_PASSWORD', 'analytics_db_password');
```

### گام سوم: راه‌اندازی پل ارتباطی و نماگیری در دیتابیس اصلی
فایل‌های مهاجرتی زیر را به صف پی‌درپی دیتابیس **اصلی** اضافه کرده و به اجرا درآورید:
1. `supabase/sql/36_marketing_fdw.sql`
   - ایجاد اکستنشن FDW و پل سرور برای فراخوانی لوکال.
   - اتصال امن به کمک نقش مپ شده `mkt_bridge_user` و ایمپورت جداول بیرونی (`events_fdw`, `campaigns_fdw`).
2. `supabase/sql/37_marketing_views.sql`
   - ایجاد هر شش نمای ماتریالیزه شده تجمیعی گزارشات:
     - `mv_traffic_overview`: بررسی کلان ترافیک‌ها در محدوده‌های ۳۰ روز و امروز.
     - `mv_funnel_by_channel`: پایش نرخ دقیق تبدیل فلو (Visit -> Register -> Premium).
     - `mv_purchase_timing`: تعیین زمان‌بندی طلایی خرید کاربران پس از عضویت.
     - `mv_retention_by_channel`: تحلیل دوره‌ای کوهورت‌ها در ماه‌های ۱، ۲، ۳ و ۶.
     - `mv_channel_roi`: سنجش کارایی مالی کانال‌ها، هزینه‌ها، درآمدها و بازدهی (ROI).
     - `mv_campaign_detail`: واکاشی جزئی کمپین‌ها جهت مقایسه‌های فنی.
3. `supabase/sql/38_marketing_cron.sql`
   - ثبت پروسه بک‌گراند آپدیت دوره‌ای و همزمان (`REFRESH MATERIALIZED VIEW CONCURRENTLY`) به کمک ابزار بومی `pg_cron`.

## ۳. امنیت و دسترسی‌ها (Security Paradigm)
- **محدودسازی دسترسی بیرونی:** تمام جداول خارجی تحلیلی بصورت رسمی در اسکیمای مجزای `marketing` محاصره گشته و از سطح دسترسی رول‌های عمومی `anon` و `authenticated` خارج شده‌اند (`REVOKE ALL`).
- **پل ارتباطی امن:** ارتباط FDW صرفاً از طریق کاربر واسط `mkt_bridge_user` برقرار گشته است تا هیچگونه دسترسی سوپریوزر به هیکل دیتابیس ریموت نشت پیدا نکند.
- **عاری بودن کلاینت از کلیدهای سرور:** لایه‌های تحلیلی پنل ادمین مستقیماً از طریق توابع ادمینِ Gateway در بستر Edge Function با دیتابیس اصلی گفتگو می‌کنند و هیچ فرانت‌اندی کلید به دستِ سرور نخواهد فرستاد.
